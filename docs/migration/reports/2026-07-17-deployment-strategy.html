초기 배포 전략: 저장소 구조 검토 & 운영계 롤아웃 계획

작성일 2026-07-17 · v2 갱신 · 전제: 기존 Nexacro 프로젝트 = 가동 중인 운영 시스템
v2 업데이트 — 실제 운영 인프라를 확인했다: Apache HTTP Server가 앞단에 있고 JBoss WAS에 Spring 애플리케이션이 올라가 있는 구조. 이전 리포트(§3)의 분기점 질문(앞단 프록시 유무)이 "있다"로 답이 나왔으므로, 전략은 더 이상 두 갈래가 아니라 하나로 좁혀진다: 같은 VM에 React 정적 리소스용 경량 서버를 새 포트로 하나 더 띄우고, Apache가 그쪽으로 라우팅만 추가한다 — JBoss·WAR는 한 줄도 안 건드린다. 저장소 구조 권장안도 이 결정에 맞춰 갱신했다(§1).

1 · 저장소 구조, 이대로 둬도 되나

✓
[v2] 별도 저장소 분리로 권장을 바꾼다 — React가 실제로 독립된 프로세스(별도 포트, 별도 배포 대상)로 뜨는 게 확정됐기 때문이다. 이전 리포트에선 "지금 단계(xfdl 원본을 읽으며 그 자리에서 포팅하는 고빈도 작업 단계)엔 모노레포가 편하다"고 했는데, 그 전제가 바뀌었다: 이제는 React가 JBoss와 완전히 다른 런타임 (Node/정적 서버 vs Java/WAR)으로, 다른 포트에서, 다른 배포 절차로 올라간다. 이 둘을 같은 저장소에 두면 CI가 "지금 커밋이 어느 쪽에 영향을 주는지" 매번 경로 필터링을 해야 하고, 배포 스크립트도 저장소 하나에서 두 산출물을 갈라 뽑아야 한다 — 굳이 감수할 이유가 없다. 저장소를 나누면 그 문제가 애초에 생기지 않는다: React 저장소는 npm run build → dist/를 새 포트의 정적 서버로 배포, JBoss 저장소는 지금처럼 mvn package → WAR 배포, 서로 완전히 독립.

단, xfdl 원본을 참고하며 화면을 하나씩 포팅하는 작업 자체는 계속 필요하다 — 새 저장소에도 docs/migration/ 같은 참고 문서나 원본 소스 경로에 대한 링크는 남겨두는 걸 권장한다(원본 레포를 서브모듈로 참조하거나, 그냥 문서에 상대 경로를 적어두는 정도로 충분하다).

실제로 점검해본 것 — 위생 상태는 이미 정상

항목상태근거
frontend/node_modules, dist 커밋 여부 정상 루트 .gitignore와 frontend/.gitignore 둘 다 node_modules/dist 제외 — 215MB node_modules는 커밋 대상에 없음.
실제 커밋될 frontend 파일 수 78개 소스코드·설정 파일만 — 빌드 산출물 섞임 없음.
Maven 빌드가 frontend를 인지하는가 미연결 pom.xml의 <build>엔 frontend 빌드 플러그인이 없다 — mvn package를 돌려도 React는 전혀 빌드되지 않는다. 이게 진짜 갭이다.
CI가 두 빌드를 같이 검증하는가 CI 없음 저장소에 CI 설정 파일 자체가 없음(GitHub Actions 등 미확인) — WAR/React 어느 쪽도 자동 빌드·테스트되고 있지 않다.

위 표는 지금 frontend/가 모노레포 안에 있어도 "빌드 산출물이 잘못 커밋되는" 문제는 없다는 뜻이다 — 그러나 두 빌드를 하나로 묶는 작업(CI든 배포 스크립트든)을 지금 시작한다면, 그 결과물은 결국 "따로 배포되는 두 산출물"이 된다. 그럴 거면 처음부터 저장소도 나누는 게 더 단순하다는 게 이번 갱신의 결론이다.

2 · 운영계 현황 감사 (저장소 + 실제 인프라)

배포 전략을 세우려면 WAR가 실제로 뭘 서빙하고 있는지, 그리고 그 앞에 뭐가 있는지부터 확인해야 했다.

항목확인 결과
패키징 war 단일 산출물 (JBoss WAS 배포 — 로컬 PoC에서 쓴 Tomcat 9는 개발 편의상 대체품이었을 뿐, 실제 운영은 JBoss)
/ 경로 소유자 DispatcherServlet이 /에 매핑돼 있지만 실제 @Controller가 하나도 없다(HelloWorld.java는 main()만 있는 미사용 클래스). 즉 지금 /는 사실상 비어 있다 — React가 /를 새로 차지해도 충돌할 기존 라우트가 없다.
실제 서빙 중인 경로 /nexacro/**, /resources/** (둘 다 servlet-context.xml의 정적 리소스 매핑) — 이 두 경로만 건드리지 않으면 기존 서비스에 영향 없음.
React↔WAR 빌드 연동 없음, 그리고 [v2] 이제 필요 없어짐 — React는 별도 저장소·별도 프로세스로 배포하기로 했으므로 WAR에 동봉할 이유가 없다.
기존 리버스 프록시/게이트웨이 확인됨 — Apache HTTP Server 가 앞단에 있고 JBoss WAS로 프록시하는 구조. §3의 분기점이 해소됐다.

3 · 분기점 해소: Apache가 이미 있다 → 경로가 하나로 좁혀짐

이전 리포트에서 던졌던 질문("Tomcat 앞에 리버스 프록시가 있는가")에 대한 답이 나왔다.

A. Apache HTTP Server + JBoss WAS. Apache가 이미 앞단에서 요청을 받아 JBoss로 넘기고 있다.

→ 확정: 기존 Apache에 라우팅만 추가 채택

  • Apache에 ProxyPass 규칙 하나 추가: / → React 정적 서버(새 포트), /nexacro/**는 지금처럼 그대로 JBoss.
  • JBoss·WAR·기존 배포 파이프라인은 한 줄도 안 건드림 — 레거시 쪽 리스크 제로.
  • 같은 VM에 React 정적 리소스를 서빙할 경량 서버 프로세스를 새 포트로 하나 추가(§4에서 구체안).
  • 롤백 = Apache 설정 되돌리기 한 줄 + graceful 리로드. WAR 재배포 불필요, 수 초 내 원복 가능.

이전 리포트의 "옵션 B(같은 WAR에 동봉)"는 폐기한다 — 앞단이 이미 있는 게 확인됐으니 JBoss를 건드릴 이유가 없다.

4 · 구체안: 같은 VM, 새 포트 + Apache 라우팅 추가

사용자가 제안한 방향 그대로다 — 같은 VM에 React용 서버를 포트만 다르게 하나 더 띄우고, Apache가 그쪽으로 서빙 자원을 넘기게 한다. 기술적으로 충분히 가능하고, 오히려 이 구성에서 가장 낮은 리스크의 선택지다.

4-1 · 새 포트에서 뭘 띄울 것인가

선택지비고
경량 Nginx (권장) 정적 파일 서빙 + SPA 딥링크 폴백(try_files $uri /index.html)을 설정 몇 줄로 해결. gzip·캐시 헤더도 기본 지원. 지금 로컬 PoC의 frontend/gateway.nginx.conf를 축소 재사용 가능(Nexacro 프록시 부분만 빼면 됨).
Node 정적 서버 (예: serve) 더 가볍게 시작할 순 있지만 SPA 폴백·캐시 헤더를 직접 옵션으로 챙겨야 하고, Node 런타임을 VM에 새로 올려야 한다는 부담이 있음.
Apache가 직접 서빙(새 프로세스 없이) Apache의 Alias + mod_rewrite(FallbackResource)로 정적 파일을 바로 서빙 — 프로세스를 하나 더 안 띄워도 되지만, 배포할 때마다 Apache 설정 영역을 건드리게 돼 레거시 앞단과 결합도가 높아짐. 사용자가 요청한 "포트 하나 더 띄우는" 방식이 배포 독립성 면에서 더 낫다.

4-2 · Apache 설정 스케치

\# 기존 설정 (이미 있을 것 — 순서 중요: 구체적인 경로가 위로 와야 함) ProxyPass /nexacro/ http://localhost:<JBoss포트>/nexacro/ ProxyPassReverse /nexacro/ http://localhost:<JBoss포트>/nexacro/ ProxyPass /resources/ http://localhost:<JBoss포트>/resources/ ProxyPassReverse /resources/ http://localhost:<JBoss포트>/resources/ \# 신규 추가 — React 정적 서버(예: 새 Nginx, 포트 3001) ProxyPass / http://localhost:3001/ ProxyPassReverse / http://localhost:3001/ \# 필요 모듈(JBoss 프록시에 이미 mod_proxy 계열을 쓰고 있다면 이미 로드돼 있을 가능성 높음) \# mod_proxy, mod_proxy_http (mod_proxy_ajp를 쓰는 구성이면 그건 JBoss 쪽 규칙에만 해당)

순서 주의: /는 모든 경로에 매치되는 가장 넓은 규칙이라 반드시 /nexacro/, /resources/ 규칙보다 아래(나중)에 와야 한다 — Apache는 등록된 순서대로 첫 매치를 적용하는 구성이 많아서, 순서가 바뀌면 /nexacro/** 요청까지 React 서버로 새어나갈 수 있다.

4-3 · 운영 특성

항목내용
TLS Apache에서 이미 종료되고 있다면 변경 없음 — Apache→React 서버 구간은 VM 내부 localhost 통신이라 평문 HTTP로 충분, 새 인증서 불필요.
세션 쿠키 Apache가 같은 도메인에서 경로만 갈라 라우팅하므로, JBoss가 굽는 세션 쿠키를 브라우저가 React 쪽 요청에도 자연히 실어 보낸다(서브도메인 분리보다 유리) — 다만 React 쪽에서 그 세션을 실제로 해석·검증하는 로직은 별도 구현 필요(이전 리포트 P3, 여전히 미해결).
프로세스 관리 새 Nginx(포트 3001)를 systemd 유닛으로 등록 — 재부팅·크래시 시 자동 재시작, 로그 로테이션은 배포된 다른 Nginx와 동일 정책 적용.
배포 흐름 React 저장소 CI → npm run build → 산출물을 VM의 정해진 경로로 배포(rsync/scp/아티팩트 다운로드) → 새 Nginx reload. JBoss 배포와 완전히 분리된 별도 파이프라인.

5 · 단계별 롤아웃 계획

  1. P0 — 배포 전 정리 (착수 전 필수)
    이번 주
    • 지금까지 작업을 main이 아닌 별도 브랜치로 커밋(모노레포에 남아있는 동안은) — main이 지금 운영 배포 기준이라면 더더욱.
    • React 소스를 새 저장소로 이전(§1) — 히스토리를 살리려면 git filter-repo(또는 git subtree split)로 frontend/ 경로만 뽑아 새 저장소로 옮기는 걸 권장, 그냥 새로 시작해도 무방.
    • 대상 VM에 새 포트(예: 3001)용 Nginx 설치 및 systemd 유닛 등록.
    • Apache 설정에 §4-2 라우팅 규칙 추가 — 스테이징에서 먼저 검증.
  2. P1 — 다크 릴리즈 (실사용자 노출 없음)
    배포는 하되 아직 아무도 못 보게
    • §4 구성대로 실제 운영 VM에 배포하되, Apache 라우팅은 아직 안 열고 내부 IP·스테이징 도메인으로만 접근 가능하게.
    • 실제 운영 JBoss와 나란히 떠 있는 상태에서 /nexacro/** 딥링크, 콜드 리로드, macOS 접속까지 다시 한번 스모크 테스트(로컬에서 이미 검증한 시나리오 그대로, 운영 인프라 위에서 재확인).
  3. P2 — 카나리 오픈
    내부 사용자 → 일부 % → 전체
    • 사내 IP 대역 또는 특정 쿠키/헤더 기준으로 먼저 열고, 문제 없으면 트래픽 비중을 단계적으로 확대.
    • 모니터링 포인트: / 응답시간·에러율, /nexacro/** 트래픽에 변화가 없는지(레거시 경로는 절대 영향받으면 안 됨).
    • 롤백 조건과 담당자를 사전에 문서화 — "무엇을 보면 되돌리는가"를 배포 당일이 아니라 지금 정해둬야 한다.
  4. P3 — 세션/인증 통합 (오픈 전 반드시 확정)
    P2 이전에 설계만이라도 완료 권장
    지난 리포트에서 "미착수"로 남겨둔 항목인데, 데모가 아니라 실제 운영 배포가 되는 순간부터는 더 이상 미룰 수 없다.
    • Apache가 경로만 갈라 같은 도메인으로 라우팅하므로 쿠키는 브라우저 기준으로 자연스럽게 공유된다(§4-3) — 서브도메인 분리보다 유리한 출발점.
    • 다만 실제 로그인/세션 체계가 있는 운영 시스템을 기준으로 "React 쪽 화면도 그 세션을 읽을 수 있는가"는 지금 저장소(로그인 기능 자체가 없는 데모)로는 검증할 방법이 없다 — 실제 운영 인증 방식(세션 쿠키? 토큰? SSO?)을 알아야 설계 가능.

6 · 확인이 필요한 질문 (운영팀 쪽에 물어봐야 함)

  • Q1 지금 운영 앞에 리버스 프록시가 있는가 — 해결됨: Apache + JBoss WAS
  • Q2 실제 운영 시스템의 로그인/세션 방식은? (세션 쿠키 / JWT / SSO 등 — P3 설계에 필요)
  • Q3 배포 파이프라인이 이미 있는가(Jenkins/GitLab CI 등), 아니면 이번에 같이 구축해야 하는가?
  • Q4 카나리 대상으로 쓸 수 있는 내부 사용자 그룹이 있는가?
  • Q5 VM의 OS/배포판(systemd 유닛 작성에 필요)과, Apache에 mod_proxy/mod_proxy_http가 이미 로드돼 있는지?
  • Q6 JBoss가 지금 Apache와 통신하는 프로토콜(mod_proxy_http인지 mod_proxy_ajp인지) — React 쪽 규칙은 http로 충분하지만 기존 규칙 형식을 참고하면 일관성 있게 작성 가능.
  • Q7 새 저장소로 옮길 때 커밋 히스토리를 유지해야 하는 요건이 있는가(사내 정책·감사 요건 등)?
!
이 리포트는 계획이다, 아직 아무것도 실행하지 않았다 저장소 분리도, Apache 설정 변경도, 새 Nginx 설치도 아직 하지 않았다 — Q2~Q7 답이 정해지는 대로 P0(§5)부터 순서대로 착수할 수 있게 준비만 해둔 상태.
