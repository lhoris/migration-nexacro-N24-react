# 화면 전환 변환 규칙 (Conversion Playbook)

> 메뉴 "정렬, 필터, 찾기"(`grid::function.xfdl`, menu_id 10100)를 React로 전환하면서 실제로 쓴
> 방법론을 사후에 문서화한 것이다. 남은 화면들을 전환할 때 같은 절차·같은 충실도 기준을
> 적용하기 위한 체크리스트로 쓴다. `docs/migration/repo-restructure-plan.md`의 "미결정 사항 3번"에
> 대한 답으로 작성됨. 이후 menu_id 10200("다양한 표현") 전환 때 Playwright MCP 기반 자동 검증
> 절차(4번)가 추가됨 — **이 문서의 절차는 모든 화면 전환에 항상 적용하는 표준 워크플로우다.**

## 0. 화면 "다 됐다"고 보고하기 전 체크리스트 (Definition of Done)

menu_id 10200("다양한 표현") 전환 때 사용자 피드백이 8라운드 오갔다. 매번 "이번엔 자동
검증 다 통과했다"고 생각한 뒤에도 계속 문제가 나온 이유를 복기하면:

| 라운드 | 사용자가 잡은 것 | 그때 검증이 놓친 이유 |
|---|---|---|
| 1 | Mask 마스킹 안 됨, Number 콤마 없음 | 스크린샷에 실제 렌더링값이 이미 보였는데, 코드 작성 시 재대조하지 않고 감으로 하드코딩함 |
| 2 | (Currency 기호·섹션 순서·요일 누락은 Playwright 텍스트 대조로 먼저 잡음 — 정상 케이스) | — |
| 3 | Checkbox/Calendar/Radio가 행별로 편집 안 됨, "왜 테이블로 만들었냐" | 텍스트만 대조했지 클릭(상호작용) 자체를 검증 항목에 넣지 않았음 |
| 4 | Calendar는 그래도 안 됨 | Checkbox/Radio/Tree 몇 개만 클릭 테스트해보고 "같은 라이브러리 기능이니 나머지도 되겠지"라고 넘겨짚음 — 실제로는 Calendar만 luxon 의존성 문제로 조용히 깨져 있었음 |
| 5 | Edit/TextArea 편집 안 됨 | 원본도 실제로 읽기 전용인 게 맞았다(버그 아님) — 그런데 이걸 처음 보고할 때 먼저 공유 안 하고 사용자가 재차 지적하게 만듦 |
| 6 | 체크박스 클릭 영역 좁음, 편집모드 전환 어색함 | 원본 충실도만 검증 항목이었지 "내가 새로 고른 위젯 자체의 UX 품질"은 검증 대상이 아니었음 |
| 7 | toggle 스위치도 별로, "제일 기본형"을 요구 | 같은 이유 — 화려한 옵션부터 고르고 나중에 단순화하는 순서였음(거꾸로 했어야 함) |
| 8 | 게이트웨이에서 원본 링크가 안 열림 | dev 서버(`:5173`)에서만 확인했고 실제 배포 토폴로지(게이트웨이 `:3000`)는 한 번도 안 열어봄 |

이걸 아래 체크리스트로 못박는다 — **전부 통과해야 사용자에게 "확인해달라"고 말할 자격이 있다.**
하나라도 건너뛰면 위 표의 라운드 중 하나를 그대로 반복하게 된다:

1. **정적 값 대조**: 원본 DOM 텍스트 + 스크린샷 속 실제 값(마스킹/포맷 등)까지 전부 내
   구현과 문자열 단위로 일치시킨다(4장 3번).
2. **에디터가 있는 컬럼/버튼은 예외 없이 전부** 클릭 → 값 변경 → 커밋까지 확인한다.
   "같은 라이브러리 기능이니 될 것"이라며 표본만 검사하고 넘어가지 않는다(4장 3-1,
   5-2번) — Calendar 하나만 개별적으로 깨져 있었던 게 이 가정 때문이었다.
3. **각 상호작용 테스트 직후 콘솔 에러/경고를 확인한다.** 페이지 로드 시점 콘솔만 보고
   끝내지 않는다 — 조용히 실패하는 라이브러리 의존성 에러(예: luxon 누락)는 클릭한
   바로 그 순간에만 콘솔에 찍힌다.
4. **원본과 의도적으로 다르게 만든 부분, 원본이 사용자 기대와 다르게 동작하는 부분은
   보고 시점에 먼저 밝히고 의견을 묻는다.** 사용자가 써보고 "왜 이건 안 되나요?"라고
   묻게 만들지 않는다.
5. **원본에 대응이 없는 내 UI 선택(에디터/포맷터 종류 등)은 원본 충실도 문제가 아니라
   순수 UX 취향 문제다 — 화려한/커스텀 옵션보다 가장 기본적인 네이티브 컨트롤을 먼저
   시도**하고, 그걸로 부족하다는 피드백이 실제로 오면 그때 더 복잡한 옵션으로 올라간다
   (거꾸로 화려한 것부터 시작해서 점점 단순화하지 않는다. 7장 참고).
6. **실제 배포 토폴로지(게이트웨이 `:3000`)에서 최소 한 번은 열어서 확인한다** — dev
   서버(`:5173`)에서만 보고 끝내지 않는다. 레거시 링크처럼 게이트웨이 프록시에 의존하는
   기능은 게이트웨이 없이는 원천적으로 검증이 불가능하다.

## 1. 원본 파악 (정독 우선, 짐작 금지)

대상 메뉴의 원본 소스를 실제로 읽고 시작한다. 최소 확인 대상:

- `spring-nexacro-N24/src/main/webapp/nexacro/<module>/<screen>.xfdl.js` — 화면 로직 전체
  (Dataset 바인딩, 이벤트 핸들러, 팝업 오픈 로직).
- 화면이 여는 팝업이나 별도 설명 화면이 있다면 그 파일도 별도로 읽는다(예:
  `GridSortPop.xfdl.js`, `GridFilterPop.xfdl.js`, `<screen>_desc.xfdl.js`) — 원본 코드베이스는
  보통 메인 화면과 이런 보조 파일을 따로 둔다.
- `<screen>.xfdl.js`의 `Format XML`/`TEXT(...)` 호출 — 그리드 헤더나 라벨이 어떤
  messageid로 다국어 처리되는지 여기서 확인한다. 이 매핑 순서를 그대로 React
  쪽 `t(key)` 호출에 옮긴다. 실제 ko/en 원문은
  `spring-nexacro-N24/src/main/webapp/nexacro/_resource_/_stringrc_/{ko,en}/stringresource.xstring.json`
  에서 그대로 뽑는다(UTF-8 BOM이 있어 `encoding='utf-8-sig'`로 읽어야 함).
- `Application.xadl.js`의 `gdsAllMenu` Dataset — 이 화면의 실제 `menu_id`, 그룹, 툴팁 원문을
  확인한다(`src/data/menu.ts`에 이미 추출된 것과 대조).
- 설명 영역에 이미지가 있으면(`_resource_/_images_/Desc/*.png`) 반드시 **직접 열어서
  본다** — Nexacro Studio 캡처 이미지 안에 실제 렌더링값(예: 마스크 적용 결과, 콤보
  옵션 등)이 박혀 있는 경우가 있고, 이게 코드만 봐서는 알 수 없는 유일한 단서일 때가
  있다(10200 화면의 Mask 컬럼 실제값 "12345\*\*\*\*\*"가 이렇게만 확인 가능했다).

목적: 실제로 동작하는 기능과, 화면에 그림으로만 있고 동작하지 않는 장식을 구분하는 것.
원본도 정적 그림일 뿐인 부분(예: 우클릭 컨텍스트 메뉴 그림)까지 새로 동작하게 만들 필요는
없다 — 원본과 동일한 수준으로만 재현한다.

**Playwright로 원본의 실제 동작(특히 마우스 드래그 등 저수준 인터랙션)을 재현할 수 없다고
해서 "확인 불가"로 단정하고 추측 구현으로 넘어가지 말 것 — 원본 컴포넌트의 런타임 소스
자체를 읽으면 답이 나오는 경우가 많다.** menu_id 10900(스마트 스크롤)에서
`grdList.fastvscrolltype`(스크롤바 드래그 중 표시되는 인디케이터)의 정확한 시각적 동작을
Playwright의 합성 마우스/포인터 이벤트로는 트랙바를 전혀 움직일 수 없어 확인하지
못했다("인디케이터가 말풍선인지 배지인지" 짐작만 하고 사용자에게 직접 드래그해보고
설명해달라고 요청함) — 그런데 사용자가 "네가 원본 기능을 제대로 이해 못 한 거 아니냐"고
되물어서 다시 보니, **`spring-nexacro-N24/src/main/webapp/nexacro/nexacrolib/component/
Grid.js`(넥사크로 컴포넌트 런타임 원본, 미니파이됐지만 변수명만 난독화됐을 뿐 로직은 그대로
읽을 수 있음)에 `set_fastvscrolltype`/`_floatingScrollRows_callback`/`_createHighlightRow`
함수가 그대로 들어있어서, 각 타입(topdisplay/centerdisplay/topbottomdisplay/
topcenterbottomdisplay/trackbarfollow)이 정확히 무엇을 하는지(스크롤 위치에 해당하는 실제
행 데이터를 테두리+그림자 있는 "떠 있는 미리보기 행"으로 보여주며, 위치는 각 타입마다
상단/중앙/하단 고정 또는 트랙바 위치를 따라 움직임) 코드 레벨에서 정확히 확인할 수
있었다**(`_floating_row_border="1px solid gray"`, `_floating_row_shadow="1px 1px 12px
gray"`, `_floating_gap=3`처럼 스타일 상수까지 그대로 노출돼 있어 픽셀 단위로 재현
가능했다). **원본 소스가 프로젝트 안에 있는데 Playwright만으로 안 된다고 포기하지 말 것 —
`nexacrolib/component/*.js`를 grep해서 관련 속성명(`set_xxx`, `_p_xxx`)을 찾아 함수 본문을
직접 읽는 것이 훨씬 빠르고 정확하다.**

## 2. 실제 데이터 추출 (가짜 데이터 금지)

원본 Dataset의 `_setContents`(또는 서버 연동이면 실제 응답 예시)에서 값을 그대로 뽑아
`src/data/<screen>RealData.ts` 같은 파일로 옮긴다. 예시 데이터를 새로 지어내지 않는다 —
행 수, 컬럼명, 실제 값(이름/날짜/금액 등)까지 원본과 동일해야 화면을 나란히 놓고
대조했을 때 신뢰할 수 있다.

파일 상단에 원본 출처를 한 줄로 남긴다(예: `sortRealData.ts`의
`// Nexacro grid::function.xfdl의 dsList Dataset(_setContents)에서 그대로 추출한 실제 데이터. 28행 전부.`).

**예외: 원본 Dataset이 `ColumnInfo`만 있고 실제 행 데이터가 없는 경우.** 서버 트랜잭션으로
채워지는 화면(예: 10500 피벗의 `svc::pivotdata`)은 이 프로젝트에 그 백엔드가 없을 수 있다 —
**추출을 포기하기 전에 원본에서 직접 그 버튼을 눌러 실제로 서버 응답이 오는지 Playwright로
확인해라.** 405/404가 뜨거나 `alert("FAILED")`가 뜨면 원본 데모 사이트 자체에서도 그 기능이
동작한 적이 없다는 뜻이다(피벗 화면에서 실측 확인: `http://localhost:3000/pivotdata` → 405).
이 경우 "원본처럼 깨진 상태를 그대로 재현" vs "클라이언트에서 목업 데이터를 만들어 실제로
동작하게 만든다" 둘 다 정당한 선택지이므로, 그 화면의 핵심 기능이 걸려 있으면(드래그 앤 드롭
피벗 조작처럼) 사용자에게 먼저 물어봐라 — `*RealData.ts` 대신 `*Mock.ts`로 파일명을 구분해
"이건 추출이 아니라 생성"이라는 걸 명확히 한다.

## 3. 기능 1:1 대조 체크리스트

옮기기 전에 원본에서 다음을 표로 뽑아둔다(코드 리뷰용, 커밋에 포함할 필요는 없음):

- 버튼/토글별: 원본 이벤트 핸들러가 실제로 하는 일
- 팝업별: 여는 조건, 입력 필드, 적용/취소 동작
- 정렬/필터 같은 상태값: 원본이 다중 선택을 허용하는지, 순서가 의미가 있는지
  (`MultiSortPopup.tsx`는 `GridSortPop.xfdl`의 `ds_item` + `btn_add/btn_delete/btn_up/btn_down`
  구조를 행 단위로 그대로 옮긴 예)
- **화면 내 섹션/영역의 실제 배치 순서**: 원본 코드의 컴포넌트 생성 순서(taborder)와
  실제 화면에 그려지는 순서(desktop/large 레이아웃의 `move()` 좌표)가 다를 수 있다 —
  10200 화면에서 "헤드 Control 표시 유형" 섹션이 taborder상으로는 맨 마지막에 생성되지만
  실제 large 레이아웃 좌표(y=318)로는 두 번째 섹션에 온 사례가 있었다. **코드만 읽고
  섹션 순서를 짐작하지 말고, 4번(Playwright 검증)에서 실제 렌더링 순서로 확정한다.**

## 4. Playwright 자동 검증 (필수 단계 — 항상 수행)

화면을 옮긴 뒤 "비슷해 보인다"로 끝내지 않는다. 아래 절차를 **항상** 거친다.

**사전 준비 (한 번만):**
- `claude mcp add playwright -- npx -y @playwright/mcp@latest`로 Playwright MCP 연결.
  (세션 도중 추가했다면 그 세션에는 안 잡히고 재연결/재시작해야 도구가 로드된다.)
- 원본 Nexacro: `spring-nexacro-N24/`에서 `mvn clean package` → sdkman Tomcat 9에 **ROOT
  컨텍스트**로 배포(웹앱 자체가 `nexacro/` 하위 구조라 ROOT로 배포해야
  `localhost:8080/nexacro/launch.html`이 README와 맞음) → 기동.
- React 앱: `spring-nexacro-N24-react/`에서 `npm run dev`(vite, 기본 5173).

**대조 절차:**
1. `mcp__playwright__browser_navigate`로 두 URL을 각각 연다 —
   원본은 `http://localhost:8080/nexacro/launch.html#<menu_id>`(URL 해시로 해당 메뉴 딥링크),
   React는 `http://localhost:5173/<path>`.
2. **스크린샷으로 먼저 구조를 훑는다** (`mcp__playwright__browser_take_screenshot`,
   `fullPage: true`). Nexacro는 자체 스크롤 컨테이너를 쓰므로 문서 `scrollTo`가 안 먹는다 —
   전체를 한 번에 보려면 `mcp__playwright__browser_resize`로 뷰포트 높이를 충분히 키운다
   (예: `height: 4200`).
3. **핵심은 스크린샷이 아니라 DOM 텍스트 직접 비교다.** `mcp__playwright__browser_evaluate`로
   `document.querySelectorAll('[id*="<GridName>."]')`처럼 원본 그리드의 실제 셀 텍스트
   (`...:text` 엘리먼트)를 뽑아 React 쪽 동일 데이터와 문자열 단위로 대조한다. 이 방식으로
   실제로 잡아낸 사례(10200 화면):
   - Mask 컬럼이 화면엔 안 보이던 마스킹(`12345*****`)을 실제로 적용하지 않고 있었음
   - Number 컬럼에 천단위 콤마 누락
   - Currency 컬럼에 원본이 실제로 붙이는 원화 기호(`￦`, U+FFE6) 누락
   - 섹션 순서가 실제 렌더링 순서와 다름(3번 체크리스트 참고)
   - 원본에만 있던 요일 접미사(예: `2020-01-01 수`)를 놓침 — 스크린샷만 봐서는 크기가
     작아 놓치기 쉬웠고, DOM 텍스트 추출로 확정함
3-1. **텍스트 대조만으로는 부족하다 — 실제로 클릭해서 편집 가능 여부까지 확인한다.**
   그리드의 각 컬럼이 행별로 실제 편집 가능한지(vs 읽기 전용 표시일 뿐인지)는 텍스트만
   봐서는 알 수 없다. `mcp__playwright__browser_click`으로 셀을 실제로 클릭해보고,
   `mcp__playwright__browser_evaluate`로 그 컬럼의 DOM에 위젯 전용 클래스가 있는지
   (예: Nexacro Grid는 `.cellcheckbox`/`.cellcombo`/`.cellcalendar`/`.cellradioitem`처럼
   컬럼별 위젯 클래스가 붙고, 읽기 전용 컬럼은 `:text`만 있다) 또는 실제 바인딩된
   Dataset 값이 클릭 전후로 바뀌는지(`grid.getBindDataset().getColumn(row, col)`)로
   확인한다. 10200 화면에서 이 방식으로: Head Control 그리드의 Checkbox/Combo/
   MultiCombo/Calendar/Radio 5개 컬럼은 행마다 실제로 개별 편집 가능한데(원본에도
   `edittype` 있음) 처음엔 읽기 전용으로 잘못 옮겼던 것, 반대로 Mask/Edit/TextArea
   3개 컬럼은 사용자가 편집을 기대했지만 원본도 실제로 읽기 전용(`edittype` 없음)인 것을
   확정했다. 트리 그룹핑도 마찬가지로 원본에 행별 개별 +/-(`.celltreeitem.treeitembutton`)가
   있는지 실제로 클릭해서 확인했다(전역 펼치기/접기 토글과는 별개 기능).
4. 확인되지 않는 값(예: 다른 언어 모드에서 실제로 어떻게 보이는지 콘솔 조작으로도 재현이
   안 되는 경우)은 **지어내지 않는다** — 확인 가능한 기본값만 반영하고, 왜 나머지를
   보수적으로 처리했는지 코드 주석에 남긴다(`Renderer.tsx`의 `formatMultiFormatDate` 참고).
   `nexacro.getApplication().langCode`를 콘솔에서 직접 바꾸는 시도는 내부 상태 불일치로
   에러가 나고, 그 상태가 `localStorage`에 저장되어 이후 새로고침까지 빈 화면이 뜨는 걸로
   이어질 수 있었다(아래 5-1 참고) — 이런 강제 상태 조작은 하지 말 것.
5. 차이를 찾으면 즉시 코드를 고치고, `npx tsc -b`로 재빌드 확인 후 Playwright로 **다시**
   같은 대조를 반복한다 — 한 번에 끝내지 않고 차이가 없어질 때까지 반복.
5-1. **Nexacro 원본 페이지를 다루다 화면이 하얗게 깨지면**: `mcp__playwright__browser_resize`로
   뷰포트를 극단적으로 크게(예: height 4200) 바꾸는 도중이나 직후에 실행하면 Nexacro
   런타임이 깨져서 이후 새로고침을 해도 계속 빈 화면만 뜨는 걸 겪었다. 복구 순서:
   ① `localStorage.clear(); sessionStorage.clear();`로 손상된 저장 상태 제거,
   ② 그래도 안 되면 `mcp__playwright__browser_tabs`로 완전히 새 탭을 열어 그쪽에서
   다시 접속(기존 탭은 닫는다), ③ 원본 Tomcat 자체를 내렸다 올리는 것도 안전한 재시도
   방법이다(서버 상태 문제는 아니었지만 확실한 클린 슬레이트가 됨). 리사이즈는 애초에
   `browser_navigate` 직후, 페이지 로드가 끝나기 전에는 하지 않는 편이 안전하다.
5-2. **Tabulator `editor: "date"`에 `editorParams.format`을 넣으면 luxon.js가 필요한데
   이 프로젝트엔 설치돼 있지 않다.** 콘솔에 `Editor Error - 'date' editor 'format' param
   is dependant on luxon.js`만 찍고 조용히 편집이 막힌다(10200 Head Control의 Calendar
   컬럼이 이렇게 죽어 있었다 — 클릭 테스트를 안 했으면 못 잡았을 버그). 저장 데이터가
   이미 ISO(`yyyy-MM-dd`) 형식이면 `format` 파라미터 자체를 빼고 기본 date input을
   쓰면 된다. 굳이 다른 포맷이 필요하면 그때 `luxon`을 실제로 설치할 것.
   에디터를 추가한 컬럼은 반드시 클릭 테스트뿐 아니라 **값을 실제로 바꾸고 커밋까지
   확인**한다(클릭만으로는 에디터가 열리는지만 확인되고, 열린 에디터가 실제로 값을
   반영하는지는 별도로 확인해야 한다) — 셀 element를 찾아 클릭 → 같은 evaluate 함수
   안에서 바로 input 값 설정 + `input`/`change` 이벤트 + `blur` 이벤트까지 한 번에
   처리해야 한다. 클릭과 값 설정을 별도의 도구 호출로 나누면 그 사이에 에디터가
   포커스를 잃어 커밋이 안 되는 경우가 있었다.
5-3. **탭/조건부 렌더링으로 마운트되는 Tabulator를, 같은 evaluate 호출 안에서 "탭 전환
   클릭"과 "그 안의 버튼 클릭"을 연달아 실행하면 데이터가 나왔다 사라지는 것처럼
   보일 수 있다.** `main.tsx`가 `<StrictMode>`를 쓰고 있어서, 컴포넌트가 처음 마운트될
   때 React가 개발 모드에서 effect를 마운트→클린업→재마운트 한 번 더 실행한다(정상
   동작, 프로덕션에서는 안 일어남) — 탭을 전환한 직후 그 탭 안의 버튼을 같은 스크립트
   안에서 바로 클릭하면, 그 클릭이 곧 사라질 첫 번째 인스턴스에 꽂혀서 잠깐 후 진짜
   인스턴스로 교체되며 데이터가 사라진 것처럼 보인다(10300 무한 스크롤 탭에서 이걸
   Tabulator/`progressiveLoad` 버그로 오인해 한참 헤맸다 — 실제로는 테스트 스크립트가
   너무 빨랐던 것뿐이었다). **해결: 탭 전환 클릭과 그 다음 상호작용을 별도의
   `mcp__playwright__browser_evaluate` 호출로 나눠서 그 사이에 자연스러운 라운드트립
   지연이 들어가게 한다** — 실제 사용자가 클릭하는 것처럼 별도 호출로 나누면 이 경합은
   재현되지 않는다.
5-4. **무한 스크롤/점진적 로딩처럼 "여러 번 반복해야 드러나는" 상호작용은 한두 번만
   눌러보고 끝내지 말고, 실제로 여러 번(수십 회) 반복 재현해야 한다.** 10300 화면에서
   스크롤을 1~2번만 테스트했을 때는 멀쩡했는데, 사용자가 실제로 200행 넘게 스크롤한
   뒤에야 "스크롤 위치가 매번 맨 위로 리셋되는" 버그를 발견했다 — Tabulator
   `addData()`가 끝나면 스크롤 컨테이너의 `scrollTop`을 되돌려주지 않고 그대로 두면
   0으로 밀리는 것이었다(반복 호출마다 매번 발생하고 있었는데, 1~2번 테스트로는
   "잘 되네" 하고 넘어가기 쉬웠다). 이후 이런 유형의 인터랙션은 Playwright로 반복문을
   돌려(`for`/`await` 루프 하나의 evaluate 안에서 20회 이상 스크롤) 매 회차의 스크롤
   위치·행 개수·첫/마지막 행 번호를 전부 기록해서 대조한다 — 끝(전체 데이터 소진)까지
   가서 정상적으로 멈추는지도 함께 확인한다. 고칠 때는 `addData` 호출 전 `scrollTop`을
   기억해뒀다가 `addData`가 반환하는 Promise가 끝난 뒤 복원하고, 동시에 중복 트리거를
   막는 재진입 가드(`loadingMore` 플래그)도 같이 넣는다.
5-5. **컬럼을 처음부터 숨겨서 그려야 할 때(localStorage 등에서 복원), 테이블 생성 후
   `column.hide()`를 호출하지 말고 컬럼 정의 자체에 `visible: false`를 넣는다.**
   10400 화면(개인화, 컬럼/행 숨기기+저장)에서 `tableBuilt` 이벤트 안에서(심지어
   `setTimeout`으로 한 틱 미뤄도) `column.hide()`를 호출하면 `column.isVisible()`은
   `false`로 바뀌는데 실제 DOM은 안 숨겨지는 경합이 있었다(`layout:"fitColumns"`의
   초기 너비 계산과 겹치는 것으로 추정 — React StrictMode 이중 마운트와는 무관하게
   프로덕션 빌드에서도 재현됨, 5-3과 다른 종류의 버그이니 헷갈리지 말 것). 컬럼
   정의에 `visible: !hiddenFields.has(field)`처럼 생성 시점에 박아 넣으면 이 경합
   자체가 없다. **반대로 사용자가 지금 우클릭 등으로 실시간 하나씩 숨길 때는
   `column.hide()`가 즉시 정상 동작한다** — 문제는 오직 "테이블을 만들면서 동시에
   여러 컬럼을 이미 숨긴 상태로 시작"하는 초기화 시점에만 있었다.
5-6. **버튼 텍스트로 `browser_evaluate`에서 요소를 찾을 때, 페이지가 지금 어느
   언어(ko/en)인지 먼저 확인해라.** 10500 화면 검증 중 `textContent.includes('Collapse
   panel')`로 못 찾아 "버튼이 안 눌린다"고 잘못 판단할 뻔한 적이 있었다 — 실제로는 직전에
   언어를 토글해서 버튼이 "패널접기"로 바뀌어 있었을 뿐이었다. 특히 하나의 세션 안에서
   한/영을 여러 번 토글하며 검증할 때, 언어 전환 버튼 자신의 `aria-label`도 함께
   번역되므로(`Switch language (KO/EN)` ↔ `언어 전환 (KO/EN)`) 그 선택자도 매번 깨진다.
   매 상호작용 전에 `document.documentElement`나 현재 보이는 라벨을 한 번 읽어 실제
   언어를 확인하고 그에 맞는 문자열로 찾거나, 언어에 무관한 `class`/`data-*` 선택자를 쓴다.
5-7. **탭마다 다른 `columns`/설정을 주면서 같은 컴포넌트 함수를 조건부로 반환하면, React는
   두 탭을 "같은 컴포넌트"로 보고 리마운트하지 않는다.** 10600 화면(대용량 데이터)에서
   `{tab === "a" ? <Foo columns={colsA}/> : <Foo columns={colsB}/>}` 형태로 짰다가, 탭을
   전환해도 Tabulator를 만드는 `useEffect`(빈 의존성 배열)가 다시 실행되지 않아 "다양한
   표현" 탭이 계속 "일반 표현"의 읽기전용 컬럼으로 렌더링되는 버그가 있었다(체크박스/콤보/
   프로그레스바가 하나도 안 보임 — Playwright로 셀 `innerHTML`을 직접 찍어봐야 알아챌 수
   있었다, 스크린샷만으론 "글자는 똑같이 나오니" 놓치기 쉽다). `key="a"`/`key="b"`를 줘서
   React가 서로 다른 인스턴스로 취급하게 하면 탭 전환마다 정상적으로 언마운트→재마운트된다.
5-8. **`layout: "fitDataFill"`(혹은 `"fitColumns"`)과 "데이터가 반복적으로 늘어나는 그리드"를
   같이 쓰지 마라 — 컬럼 폭을 셀 텍스트 실측으로 재계산하는 작업이 데이터가 바뀔 때마다
   전체 행을 다시 훑는다.** 11300 화면(분할 조회, 10,000행씩 10번 `addData`로 총 10만행을
   쌓는 시나리오)에서 이걸 그대로 뒀다가 배치가 늘어날수록 매 `addData` 호출이 점점 느려져
   9~10번째 배치쯤에서 브라우저 탭이 30초 넘게 완전히 응답 없이 멈췄다(Playwright
   `browser_wait_for`/`browser_console_messages` 전부 타임아웃 — 탭을 강제로 닫고 새로
   열어야 했음). 컬럼에 이미 고정 `width`를 줬다면 `layout`을 아예 생략하는 게 안전하다.
   더 근본적으로는, 가상 스크롤 그리드에 "배치마다 반복 `addData`"를 거는 패턴 자체가
   총 행 수가 커질수록(이 케이스는 10만) 위험하다 — 카운터(UI 표시용 진행률)는 배치마다
   올리되, 실제 `table.setData()`는 전체 배치를 다 모은 뒤 **한 번만** 호출하는 쪽이
   안전하다(실제로 적용한 수정).
5-9. **Tabulator를 만드는 `useEffect`와 `setColumns`/`setData`를 거는 `useEffect`를 분리하면,
   React StrictMode의 마운트→클린업→재마운트 사이에서 "Cannot read properties of null
   (reading 'firstChild')" 같은 에러가 날 수 있다** — Tabulator가 내부 DOM을 다 갖추기 전에
   `setColumns`가 호출되는 경합이다. 10700 화면(퀀텀 그리드)에서 처음엔 Pivot처럼 "테이블은
   한 번만 만들고 데이터/컬럼만 별도 effect에서 갱신"하는 패턴을 그대로 썼다가 이 에러를
   만났다 — Pivot은 `facts`가 `null`일 때 두 번째 effect가 아무 것도 안 하고 리턴하는 가드가
   있어서 경합이 안 드러났을 뿐, 근본적으로 같은 위험이 있는 패턴이었다. 데이터가 몇백 행
   수준으로 작고 매번 새로 그려도 부담 없는 화면이라면, effect를 분리하지 말고 상태가 바뀔
   때마다 **테이블 자체를 destroy 후 새로 생성**(컬럼·데이터를 생성자에 바로 넣어서)하는
   쪽이 더 단순하고 이런 경합에서 안전하다.
5-10. **원본이 "버그처럼 보이는 값"을 항상 똑같이, 에러 없이 보여준다면 그건 지킬 대상인
   원본의 실제 동작이지 임의로 고칠 대상이 아니다.** 10700 화면에서 그룹핑된 트리의 리프
   행(실제 데이터 행)이 항상 "이름(0)"으로 표시되는 걸 발견했다 — 원본 소스
   (`fnSetSubSumText`)를 읽어보니 리프 레벨에서 배열 인덱스가 범위를 벗어나 first_name으로
   폴백되는 구조였다. 짐작으로 "의도한 동작 같지 않으니 빈 문자열로 고치자"라고 판단하지
   않고, Playwright로 1단/2단 그룹 모두 실측(원본이 실제로 항상 이렇게, 에러 없이 일관되게
   보여줌을 확인)한 뒤 그대로 재현했다 — 코드만 읽고 "이건 버그다"라고 넘겨짚지 말고,
   실제 화면에서 재현되는 걸 먼저 확인하라는 원칙(1번 섹션)이 여기서도 그대로 적용된다.
5-11. **Tabulator 내장 `row.freeze()`(고정 행)는 가로 스크롤이 있는 그리드에서 신뢰하지
   마라.** 10800 화면(틀 고정)에서 겪음 — 컬럼이 많아(11개, 1510px) 헤더 콘텐츠가
   컨테이너보다 넓은 상황에서 `row.freeze()`를 쓰면 내부 `frozen-rows-holder`가 헤더
   콘텐츠 바로 오른쪽(가로 스크롤 밖)에 위치해버려 전혀 안 보인다(`redraw()`로도 안
   고쳐짐). 대안으로 CSS `position:sticky`를 행에 직접 걸어봤지만, 고정된 행과 스크롤
   중인 일반 행이 같은 목록 안에 섞여 있어서 특정 스크롤 위치에서 우연히 픽셀 좌표가
   겹치며 서로 다른 행 내용이 뒤섞여 보이는 문제가 있었다(둘 다 Playwright로 DOM 좌표를
   직접 찍어서 원인 확인). **최종 해법: 고정할 행만 담은 두 번째(스크롤 없는) Tabulator
   인스턴스를 메인 그리드 "위"에 별도로 얹고, 메인 그리드의 가로 스크롤 이벤트만 그 두
   번째 인스턴스에 동기화한다.** 엑셀의 틀고정과 원리가 같고, 두 그리드가 물리적으로
   분리돼 있어 겹침 문제 자체가 없다. 컬럼 고정(`frozen` 속성)은 이 문제가 없어 그대로
   써도 된다 — 문제는 오직 "행" 고정 쪽이었다.
5-12. **Tabulator Row/CellComponent의 `getPosition()`은 1부터 시작한다(0이 아니다).**
   10800 화면에서 이걸 몰라서 기본 선택 행(배열 인덱스 2)을 저장할 때 `getPosition()`
   값을 그대로 썼다가 실제로는 인덱스 2가 아니라 3(포지션 값 그대로)이 저장되어, "행
   고정"을 누르면 항상 의도한 것보다 한 행 더 고정되는 버그가 났다(Playwright로 `getRows()
   [i].getPosition()`을 직접 찍어 1부터 시작함을 확인). `getRows()` 같은 0-based 배열의
   인덱스로 쓰려면 `getPosition() - 1`을 해야 한다.
5-13. **"화면 하나 완성" ≠ "실사용에서 안정적"이다 — 사용자가 직접 눌러보고서야 드러난
   버그 두 개를 10800 화면에서 겪었다.** Playwright로 각 버튼을 한 번씩 눌러보고 콘솔
   에러 없음까지 확인한 뒤 사용자에게 넘겼는데, 사용자가 실제로 써보니 (a) 헤더가
   고정 행 아래로 밀려 내려가 보였고 (b) 클릭 반응이 눈에 띄게 느렸다. 원인:
   - **레이아웃 순서**: 고정 행 전용 미니 그리드를 메인 그리드 "아래"에 별도 마운트
     div로 넣어놨는데, 메인 그리드가 자기 헤더를 계속 갖고 있다 보니 시각적으로
     [고정 행] → [메인 헤더] → [스크롤 본문] 순서가 되어 헤더가 중간에 끼어 보였다.
     내가 한 번씩 눌러본 스크린샷에서는 이게 "이상하다"고 느껴지지 않았다 — 이런
     레이아웃 순서 문제는 정적 스크린샷 한 장보다, 실제로 헤더가 "어디 있어야 하는지"
     기대치를 갖고 봐야 눈에 띈다.
   - **성능**: 컬럼 고정을 컬럼 11개에 대해 `column.updateDefinition()`을 11번 따로
     호출했더니 500행 그리드에서 매 호출마다 전체 재렌더링이 일어나 버튼 클릭 한 번에
     400ms 넘게 걸렸다(직접 실측 전엔 몰랐음 — "잘 눌러보니 되네" 수준으로는 안 느껴져도
     사용자는 바로 느낀다). `updateDefinition`을 컬럼별로 반복 호출하지 말고
     `table.setColumns(전체컬럼배열)`로 한 번에 갈아끼우면 20배 이상(440ms→21ms)
     빨라진다 — 여러 컬럼 속성을 한꺼번에 바꿀 땐 항상 `setColumns` 한 번을 우선 고려할 것.
   - **교훈**: 클릭 한 번 해보고 "동작하네" 하고 넘기지 말고, `performance.now()`로 실제
     걸린 시간을 재보고(체감상 100ms 넘으면 사용자가 "느리다"고 느낀다), 레이아웃이
     바뀌는 기능은 스크린샷을 "원본이 기대하는 배치와 일치하는가" 기준으로 대조할 것.
5-14. **"두 번째 그리드로 겹침을 피한다"는 해법 자체가 새 버그(레코드 중복)를 만들 수
   있다 — 데이터도 두 그리드 사이에서 겹치지 않게 나눠야 한다.** 5-11에서 고정 행을
   별도 그리드로 분리해 시각적 겹침은 해결했지만, 메인 그리드에 여전히 "전체 500행"을
   그대로 넣어놨더니 고정 그리드(0~N행)와 메인 그리드(항상 0행부터 시작)가 스크롤
   맨 위에서 같은 레코드를 두 번 보여줬다(사용자 리포트: "동일한 레코드가 보임" — 원본은
   그리드가 하나뿐이라 이런 중복이 없다). 메인 그리드 데이터를
   `records.slice(frozenCount)`로 고정된 만큼 잘라내야 원본과 같은 "이어지는 하나의
   목록"처럼 보인다. 이렇게 데이터를 나누면 "선택된 셀이 몇 번째 행인가"도 두 그리드
   기준이 서로 달라지므로(고정 그리드는 0부터, 메인 그리드는 frozenCount만큼 밀려서
   시작), 절대 인덱스 ↔ 그리드별 상대 인덱스 변환을 양방향으로(선택 저장 시, 복원 시)
   다 챙겨야 한다 — 그리고 복원 시 "선택된 셀이 있는 쪽 그리드"의 `tableBuilt`를
   기다려야지, 아무 그리드의 `tableBuilt`에서나 찾으면 상대방 그리드가 아직 준비되기
   전이라 선택 표시가 사라지는 버그가 또 생긴다(실제로 겪음).
5-15. **Tabulator 기본 테마의 `frozen` 컬럼 시각 효과(흰색 2px 굵은 경계선)가 원본과
   다르면 사용자에게는 "정렬이 이상하다"처럼 보일 수 있다.** 원본 소스(`fnColFixL`)는
   고정된 컬럼의 헤더 배경색만 살짝 바꾼다(`setCellProperty("head", i, "background",
   "#48399A")`) — 정렬이나 굵은 경계선 같은 건 전혀 안 건드린다. Tabulator의 기본
   `.tabulator-frozen` 스타일이 이질적으로 두드러지면, `getComputedStyle`로 재 봤을 때
   `text-align`이 실제로 안 바뀌었더라도 사용자 눈에는 "내용이 오른쪽으로 쏠렸다"처럼
   읽힐 수 있다 — 코드로 재현이 안 된다고 사용자가 잘못 봤다고 넘기지 말고, 원본이
   실제로 하는 최소한의 변화(헤더 배경색만)로 시각 효과를 맞추면 이런 종류의 위화감
   자체가 없어진다.
5-16. **Tabulator는 `selectableRows`를 명시적으로 설정 안 해도 기본값이 `"highlight"`라,
   행 전체에 마우스오버 하이라이트(`.tabulator-selectable:hover`)가 자동으로 걸린다 —
   원본이 셀 단위 선택 그리드(`grdList.set_selecttype("cell")`)라면 이 기본값이 "행
   선택형"처럼 보이게 만드는 원인이다.** 10800 화면에서 사용자가 "넥사크로 원본은 행이
   아니라 셀 선택형 모드인 것 같다"고 지적할 때까지 이 기본 동작을 그냥 두고 있었다.
   `selectableRows: false`로 꺼야 행 단위 호버 효과가 사라지고, 우리가 이미 구현해둔
   셀 단위 `.fp-selected-cell` 강조만 남아 원본과 같은 상호작용 모델이 된다. **원본
   Nexacro 그리드가 셀/행 중 어느 선택 모드인지는 `set_selecttype()` 호출로 소스에
   이미 나와 있으니, Tabulator 쪽 그리드를 만들 때 그 값에 맞춰 `selectableRows`를
   명시적으로 설정할 것 — 기본값에 맡기지 말 것.**
5-17. **Tabulator 내장 `selectableRange`(엑셀 스타일 셀 테두리 하이라이트)는 "얼린
   컬럼이 2개 이상"이면 라이브러리 자체가 "예측 불가 동작"이라고 경고하고, 실제로도
   경고대로 깨진다(직접 겪음 — 헤더가 흐려지고 이상한 줄이 생김).** 10800 화면에서
   selectableRows:false 다음 단계로 "그럼 셀 하이라이트는 selectableRange로 하면
   되겠다"고 붙였다가 이 경고를 봤다. 코드: `tabulator.js`의
   `updateHeaderColumn()`이 `frozenCols.length > 1`이면 무조건 경고를 찍고, 실제로
   여러 컬럼을 동시에 얼리는 화면(좌측열고정처럼)에서 헤더가 흐려지는 등 시각적으로도
   깨졌다. 이런 화면은 `selectableRange`를 쓰지 말고, 직접 만든 CSS 클래스(테두리
   `outline`, `background` 아님 — Tabulator 테마가 `.tabulator-cell{outline:none}`을
   이미 정의해서 같은 특이도로는 안 먹힌다. `.tabulator-cell.내클래스`처럼 클래스를
   하나 더 붙여 특이도를 올릴 것)로 같은 느낌만 재현하는 편이 안전하다. **Tabulator의
   "여러 기능을 동시에 켜도 되는지"는 콘솔 경고 문구를 무시하지 말고 반드시 확인하고
   넘어갈 것 — 경고가 그냥 경고가 아니라 진짜 버그를 예고하는 경우가 있다.**
5-18. **Tabulator는 셀 자체에 배경색이 없고 행(`.tabulator-row`)의 배경이 비쳐 보이는
   구조라, `position:sticky`로 고정한 셀은 스크롤로 지나가는 다른 컬럼과 화면상 같은
   자리를 차지하면서 그 뒤가 비쳐 보일 수 있다 — 헤더든 본문 셀이든 마찬가지다.**
   10800 화면에서 두 번 겪었다: ① 고정된 컬럼의 헤더에 옅은 배경색(`rgba(...,0.14)`,
   원본의 "헤더 색만 살짝 바뀜" 재현용)을 반투명으로 줬더니 가로 스크롤 시 그 뒤로
   지나가는 헤더 텍스트가 비쳐 겹쳐 보였다(사용자 리포트: "옆으로 갈 때 헤더 텍스트가
   겹쳐보인다") — 완전 불투명한 솔리드 색으로 바꿔 해결. ② 셀 마우스오버 하이라이트를
   추가하면서 또 반투명 `rgba`를 썼다가 같은 증상이 재발했다 — 이때는 CSS
   `background-color`가 값 하나라 "불투명 베이스 + 반투명 오버레이"처럼 레이어링되지
   않고 그냥 덮어써진다는 걸 몰랐던 게 원인이었다(고정 셀에 미리 넣어둔 불투명
   배경 규칙이 호버 시 통째로 무효화됨). 흰 배경과 미리 섞어 계산한 완전 불투명
   색(`#e5e9f7` 같은)으로 바꿔야 진짜로 해결된다. **고정(sticky) 요소에 배경색을 줄
   때는 절대 `rgba`/반투명을 쓰지 말 것 — 화면에 "보기엔 옅은 색"이어야 해도 값
   자체는 완전 불투명해야 한다.**
5-19. **Tabulator "modern" 테마 자체가 `.tabulator-header{padding-left:10px}`를 기본으로
   깔아둔다(우리가 추가한 스타일이 아니라 라이브러리 원본 CSS) — 헤더 컬럼이 본문
   컬럼보다 항상 10px 오른쪽으로 밀려 보인다.** 10800 화면에서 사용자가 "헤더랑 본문이랑
   열이 안 맞는 것 같다"고 지적해서 실측(`getBoundingClientRect()`로 헤더/본문 같은
   컬럼의 x좌표 비교)했더니 정확히 10px 오프셋이었다. **이건 이 화면만의 문제가
   아니라 Tabulator를 쓰는 화면 전부(퀀텀 그리드 등)에 똑같이 있던 문제라서, 화면별
   CSS가 아니라 `src/styles/shell.css`(전역으로 로드되는 파일)에 `.tabulator-header{
   padding-left:0 !important}` 한 줄로 고쳤다 — 이미 나온 화면들도 다시 열어 헤더/본문
   정렬과 콘솔 에러 없음을 재확인.** 그리드 라이브러리를 쓰는 화면에서 "약간 어긋나
   보인다"는 피드백을 받으면, 그 화면만의 문제인지 먼저 다른 화면에서도 같은 증상이
   재현되는지 확인하고, 재현되면 화면별로 고치지 말고 전역 스타일 파일에서 한 번에
   고칠 것.
5-20. **같은 테마에 비슷한 종류의 기본값이 하나 더 있었다 —
   `.tabulator .tabulator-header .tabulator-frozen.tabulator-frozen-left{padding-left:
   10px}`, 고정된 컬럼 그룹의 "경계 컬럼"(왼쪽 고정의 마지막 컬럼) 헤더에만 10px 패딩을
   더 준다.** 사용자가 5-19를 고친 뒤에도 "좌측열고정 누르면 그 경계 컬럼(예:
   Last name)의 헤더 텍스트만 오른쪽으로 밀려 보인다"고 다시 지적해서 발견 — 본문 셀은
   이 규칙 대상이 아니라(계산해보니 `padding-left:4px`로 정상) 안 밀리는데 헤더만
   밀려서 눈에 띄었다. 오른쪽 고정(`tabulator-frozen-right`)엔 이 규칙 자체가 없어
   좌우가 비대칭인 것도 확인. 5-19와 같은 곳(`src/styles/shell.css`)에
   `.tabulator-header .tabulator-frozen.tabulator-frozen-left{padding-left:0
   !important}`로 같이 고쳤다. **테마 기본값 버그를 하나 고쳤다고 "이 계열은 끝났다"고
   단정하지 말 것 — 같은 원인(고정 컬럼 관련 테마 CSS)에서 파생된 형제 버그가 더 있을
   수 있다는 걸 보여준 사례.**
5-21. **행+열 동시 고정 시, 고정 행 밴드(별도 Tabulator 인스턴스)가 자기 몫의 가로
   스크롤바를 따로 만들어서 화면에 스크롤바가 2개로 보였다.** 원본 Nexacro는 그리드가
   물리적으로 하나뿐이라 가로 스크롤도 하나인데, 우리는 고정 행을 별도의(스크롤 없는
   의도였던) Tabulator 인스턴스로 분리해뒀지만 그 인스턴스도 컬럼 폭 합이 뷰포트보다
   넓으면 `.tabulator-tableholder`가 `overflow-x:auto` 기본값 그대로라 자기 몫의
   스크롤바를 만들어버린다는 걸 놓쳤다. 사용자 리포트: "행열고정 눌렀을때... 넥사크로
   원본에는 스크롤이 한개뿐인데... 고정행에도 좌우스크롤이 있고... 아래 그리드에도
   좌우스크롤이 있어." `getBoundingClientRect`/`scrollWidth`·`clientWidth` 비교로 고정
   밴드 쪽도 메인 그리드와 똑같이 `scrollWidth(1510) > clientWidth(576)`임을 확인해
   원인을 특정했다. 고정 밴드의 `.tabulator-tableholder`에 `overflow-x:hidden
   !important`를 줘서 스크롤바 자체를 없애고, 기존에 있던 메인→고정 밴드
   `scrollHorizontal` 동기화 코드(`holder.scrollLeft = left`)는 `overflow:hidden`
   상태에서도 그대로 동작함을 확인(Playwright로 메인 그리드를 실제로 스크롤시켜
   고정 밴드 컬럼이 같은 위치로 따라오는지, 콘솔 에러 없는지 dev 서버·게이트웨이(:3000)
   양쪽에서 재확인). "스크롤 가능한 요소를 시각적으로만 숨기고 싶을 때는 `overflow:
   hidden`으로 스크롤바만 지우고 `scrollLeft`는 계속 JS로 제어"하는 패턴을 재사용할 것.
5-22. **전역 Tabulator 오버라이드의 선택자 문자열이 Tabulator 자체 CSS의 선택자와
   완전히 똑같으면, `npm run build`(esbuild CSS 압축)가 `!important`를 무시하고 통째로
   버릴 수 있다 — dev 서버(HMR, 비압축)에서는 정상으로 보여서 이 차이를 못 알아챈다.**
   10900 화면(스마트 스크롤) 작업 중 발견한 새 테마 기본값(모든 행 첫 컬럼에 10px 색상
   액센트 바 — `.tabulator-row .tabulator-cell:first-child{border-left:10px solid
   #3759d7}`, 짝수 행 `#627ce0`)을 고치려고 shell.css에 **선택자를 그대로 복사해**
   `.tabulator-row .tabulator-cell:first-child{border-left:1px solid #fff !important}`를
   추가했더니 — dev 서버에서는 액센트 바가 사라져 정상으로 보였지만, `npm run build` 후
   `dist/assets/*.css`를 grep해보니 **내가 추가한 규칙 자체가 파일에 없었다**(Tabulator
   원본 규칙만 그대로 남아 있었고, 게이트웨이(:3000)에서는 액센트 바가 그대로 보임).
   원인: esbuild CSS 압축기가 완전히 같은 선택자 텍스트를 가진 규칙들을 값이 다를 때도
   병합하면서 소스 순서상 나중 것(Tabulator 자체 CSS가 화면 컴포넌트별 import 때문에
   shell.css보다 번들 뒤쪽에 옴)을 채택하고 `!important`를 무시해버린 것으로 보인다(같은
   값을 가진 여러 선택자를 콤마로 묶는 병합은 정상 동작 확인 — 5-19/5-20의 두 padding-left
   규칙은 실제로 `.tabulator-header,.tabulator-header .tabulator-frozen.
   tabulator-frozen-left{padding-left:0!important}`로 안전하게 합쳐져 있었다; 문제는 값이
   다른데 선택자 텍스트가 같을 때만 발생). 조상 선택자(`.shell`)를 붙여 특이도를 올리는
   동시에 선택자 문자열 자체를 Tabulator 원본과 다르게 만들어(`.shell .tabulator-row
   .tabulator-cell:first-child`) 해결 — 재빌드 후 `dist/assets/*.css`에서 그 문자열이
   실제로 존재하는지 grep으로 확인하고 나서야 게이트웨이에서도 고쳐진 걸 볼 수 있었다.
   **교훈: Tabulator 전역 오버라이드를 shell.css에 추가할 때는 (1) 선택자에 우리 앱만의
   조상 클래스를 최소 하나 섞어 라이브러리 원본과 텍스트가 겹치지 않게 하고, (2) dev
   서버에서 "고쳐진 것처럼 보인다"로 끝내지 말고 `npm run build` 후 실제
   `dist/assets/*.css`를 grep해서 그 규칙이 살아있는지 확인한 다음에 게이트웨이에서
   재검증한다.** dev와 prod가 이렇게 조용히 갈라질 수 있다는 뜻이라, 이번에 고친 두
   padding-left 규칙도 이번 빌드에서 다시 grep으로 재확인해 무사함을 확인했다.
5-23. **스크롤되는 컨테이너(`overflow:auto`) 안에 `position:absolute` 오버레이 요소를
   자식으로 넣으면, 그 요소도 콘텐츠와 함께 스크롤되어 화면 밖으로 사라진다 — "뷰포트
   기준으로 고정된 오버레이"를 만들려면 스크롤 안 되는 조상에 붙여야 한다.** 10900
   화면(스마트 스크롤)의 "떠 있는 미리보기 행"을 처음엔 `.tabulator-tableholder`(실제
   스크롤되는 요소)의 자식으로 추가하고 `position:absolute; top:3px`를 줬는데, 인라인
   스타일은 정확히 원하는 값으로 찍혀 있는데도(`display:flex`, `top:3px` 등) 화면에 전혀
   안 보였다 — `getBoundingClientRect()`로 찍어보니 좌표가 전부 `{0,0,0,0}`으로 나와서야
   원인을 알았다: 스크롤 컨테이너 안의 absolute 자식은 "뷰포트의 위에서 3px"가 아니라
   "스크롤되는 콘텐츠 원점에서 3px"에 위치하고, 스크롤이 많이 내려가면 그 콘텐츠 원점
   자체가 화면 훨씬 위로 밀려나면서 요소도 함께 화면 밖으로 사라져 버린다. 해결: 오버레이
   요소를 스크롤 컨테이너가 아니라 그 바깥의, 스크롤 안 되는 조상(`table.element`, 즉
   `.tabulator` 루트 — 헤더와 tableholder를 함께 감싸고 그 자체는 스크롤되지 않음)에
   붙이고, 헤더 높이만큼을 top 오프셋에 더해 보정했다. **교훈: 오버레이를 "화면(뷰포트)
   기준 위치"에 고정하고 싶을 때 `position:absolute`만 쓰고 스크롤 컨테이너 안에 넣으면
   틀린 결과가 나온다 — 반드시 스크롤 안 되는 조상에 붙이거나 `position:fixed`/
   `position:sticky`를 검토할 것. 스타일 인라인 값이 "의도한 대로" 찍혀 있어도
   `getBoundingClientRect()`로 실제 렌더링 좌표를 확인하기 전까지는 "적용됐다"고 믿지
   말 것.**
6. 검증에 쓴 스크린샷/스냅샷 파일은 커밋 대상이 아니다(`.gitignore`의 `.playwright-mcp/`가
   커버; 루트에 저장된 임시 png는 작업 후 직접 지운다).
7. 이 자동 대조가 끝난 뒤에만 사용자에게 "최종 확인해달라"고 요청한다 — 사용자가 직접
   보기 전에 이미 코드 레벨 검증은 끝나 있어야 한다. **단, 이 절차를 거쳤어도 사용자의
   실제 사용 경험에서 나오는 피드백(상호작용 안 됨, 시각적으로 어색함 등)을 가볍게 보지
   않는다** — 10200 화면에서 텍스트/데이터 대조까지 다 통과했는데도 사용자가 직접
   써보고서야 Checkbox/Calendar/Radio가 행별로 편집이 안 된다는 걸 잡아냈다. 클릭
   테스트(3-1)를 텍스트 대조와 같은 비중으로 항상 같이 할 것.

## 5. 다국어(i18n) 연동 패턴

- 화면 텍스트는 `useLanguage()`의 `t(key, fallback?)`로 가져온다. `key`는 원본의
  실제 messageid를 그대로 쓴다(새로 짓지 않음).
- 언어 전환 시 다시 계산해야 하는 값(그리드 컬럼 타이틀 등)은 `useMemo(..., [lang])`로
  래핑하고, Tabulator처럼 외부 라이브러리 인스턴스가 있는 경우 `setColumns`로 통째로
  재생성하지 말고 `column.updateDefinition()`으로 필요한 부분만 갱신한다(정렬·스크롤
  상태가 날아가는 것을 막기 위함 — `SortFilterFind.tsx`의 언어 전환 `useEffect` 참고).

## 6. 레거시 폴백 링크

React로 전환된 화면에도 원본 Nexacro 화면으로 가는 링크를 상단에 남긴다
(`/nexacro/launch.html#<menu_id>`, `SortFilterFind.tsx`의 `sff-legacy-link-row` 참고).
전환 검증 기간 동안 두 화면을 언제든 비교할 수 있게 하기 위함이다. 이 링크를 걷어내는
시점은 `v4-legacy-retired` 마일스톤(레거시 완전 폐기) 이후다.

## 7. 파일 배치 컨벤션

- 화면 컴포넌트: `src/routes/converted/<ScreenName>.tsx` + 같은 폴더에 전용 CSS.
- 화면 전용 팝업: 같은 폴더에 별도 컴포넌트 파일로 분리(`ColumnFilterPopup.tsx`,
  `MultiSortPopup.tsx`, `FindPanel.tsx`처럼) — 팝업 하나당 파일 하나. 팝업이 아니라 같은
  화면 안의 섹션(예: 10200의 셀 표시 유형/트리 그룹핑/...)이면 한 파일 안에 섹션별
  컴포넌트 함수로 나눈다(`Renderer.tsx` 참고) — 파일을 쪼개지 않는다.
- 실데이터: `src/data/<screen>RealData.ts`.
- **기본값은 Tabulator(ADR-006)다 — "그리드" 화면을 일반 테이블 마크업으로 만드는 걸
  예외로 취급하고, 그 예외가 정당한지 먼저 의심할 것.** 10200 화면에서 처음엔 "행이
  몇 개 안 되고 위젯이 다양하니 테이블이 더 간단하다"고 판단해 전부 테이블로 만들었다가,
  사용자가 실제로 써보고서야 문제를 지적했다 — 일반 테이블은 원본이 실제로 지원하는
  행별 개별 편집(체크박스 토글, 콤보 선택, 캘린더 입력, 트리 행별 +/- 등)을 그대로
  재현하기 어렵고, 결과물이 "그리드 흉내만 낸 정적 화면"처럼 보인다. 여러 행의 실데이터를
  다루는 섹션(멀티 포맷, 표현식, 헤드 Control, 트리 그룹핑 등)은 데이터가 2~5행뿐이어도
  Tabulator로 만든다 — 트리는 `dataTree`/`dataTreeChildField` 옵션으로 원본의 개별
  행 +/- 토글을 기본 제공받고, 콤보/캘린더도 `editor: "list"/"date"`로 실제 행별 편집이
  자연스럽게 된다.
  일반 테이블 마크업은 정말로 예외적인 경우에만 — 10200의 "셀 표시 유형" 섹션처럼 원본
  자체가 Dataset **한 행**을 Format XML의 서로 다른 Row 밴드 두 개로 나눠 17개의 완전히
  다른 위젯 타입을 보여주는 카탈로그일 때, 즉 애초에 "여러 행의 동일 컬럼 구조"라는
  Tabulator의 전제 자체가 성립하지 않을 때만 쓴다.
- **원본에 대응이 없는 내 UI 선택(에디터/포맷터 종류)은 가장 기본적인 네이티브 컨트롤부터
  시작한다 — 화려한 라이브러리 formatter를 먼저 고르고 나중에 단순화하지 않는다.**
  Boolean 컬럼(체크박스류)에서 `editor: "tickCross"`(네이티브 체크박스로 순간 전환되는
  어색함) → `formatter: "toggle"`(iOS 스타일 스위치, 그래도 별로라는 피드백) →
  최종적으로 **순수 `<input type="checkbox">`를 커스텀 formatter로 그리고
  `cellClick`으로 셀 전체를 클릭 영역으로 넓히는 방식**에 정착했다(`Renderer.tsx`의
  `checkbox` 컬럼 참고) — 처음부터 이 "제일 기본형"으로 시작했으면 두 번의 재작업을
  피할 수 있었다. 다음 화면에서 비슷한 boolean/선택형 컬럼을 만들 때는 이 순서
  (네이티브 → 그래도 부족하면 그때 커스텀)를 기본으로 삼는다.
- 전환 완료 후 `src/data/menu.ts`에서 해당 메뉴 항목의 `target`을 `"nexacro"` →
  `"react"`로 바꾸고, `src/App.tsx`의 `CONVERTED_SCREENS`에 `menu_id -> 컴포넌트` 매핑을
  추가한다.
- **Nexacro 그리드 기능을 Tabulator에 커스터마이징해 추가한 것 중, 다른 화면에서도 또
  필요할 만한 것은 화면 전용 파일에 묻어두지 말고 `src/lib/tabulator/`에 재사용 가능한
  훅/유틸로 뽑는다.** 이 프로젝트의 실질적인 작업은 "화면을 React로 옮기는 것"이자
  동시에 "Nexacro 그리드가 하던 걸 Tabulator 위에 하나씩 재현해 쌓는 것"이기도 하다 —
  10800(틀 고정)에서 여러 차례의 사용자 피드백 끝에 다듬어진 "고정 행 전용 두 번째
  Tabulator 인스턴스 + 가로 스크롤 동기화" 로직을 `useFrozenRowsGrid`
  (`src/lib/tabulator/useFrozenRowsGrid.ts`)로 뽑아낸 게 첫 사례. 판단 기준: 사용 사례가
  하나뿐인 시점에 미리 뽑으면 인터페이스를 잘못 추측할 위험이 있으니, 남은 화면 목록을
  보고 같은 요구가 또 나올 게 거의 확실한 경우(예: 행/열 고정, 셀 선택형 상호작용 등
  Nexacro 그리드의 표준 기능)만 그 자리에서 바로 뽑고, 애매하면 실제로 두 번째 사용처가
  생겼을 때 두 사례를 보고 뽑는다.
- **그리드가 있는 화면은 `.work-card`의 `max-width` 캡을 없앤다(`max-width: none`) — 원본
  divWork는 창 폭에 맞춰 작업 영역을 거의 꽉 채우지, 고정된 좁은 폭이 아니다.** 10900
  화면(스마트 스크롤)에서 사용자가 "오른쪽에 여백이 많이 남는데 왜 공간을 안 쓰냐, 원본도
  그러냐"고 물어서 원본 `divWork`를 실측했더니(990px, 창 폭 - LNB - 여백만큼을 거의
  그대로 사용) 우리처럼 화면마다 임의의 px 캡(900~1200px)을 씌워 좁혀두는 방식이 아니었다.
  **더 심각한 문제를 이 과정에서 발견**: 이미 있던 캡 규칙 대부분이
  `.work-card:has(.xx-card)`처럼 마커 클래스를 work-card **자기 자신**에 붙여둔 채
  `:has()`로 검사하고 있었는데, `:has()`의 인자는 후손(descendant)만 검사하지 자기 자신은
  검사하지 않는다 — 즉 이 캡들은 **처음 추가된 순간부터 단 한 번도 실제로 적용된 적이
  없었다**(FreezePanes/LargeData/Pivot/QuantumGrid/SplitLookup/SmartScroll 6개 화면,
  `getComputedStyle()`로 직접 확인). 우연히 다른 진짜 후손 클래스(`.xx-section`,
  `.pz-grid-mount` 등)를 같이 나열해 뒀던 4개 화면(Pagination/Personalization/Renderer/
  SortFilterFind)만 정상 작동하고 있었다. `.work-card.xx-card`(같은 엘리먼트면 그냥
  붙여쓰기)로 선택자를 고치고 나서야 실제로 적용됨을 확인했다. **교훈: `:has()`에 넘기는
  클래스가 대상 엘리먼트 자기 자신에 있는 클래스인지 진짜 후손의 클래스인지 헷갈리지
  말 것 — 같은 엘리먼트면 `:has()` 없이 그냥 `.a.b`로 쓰고, CSS 규칙을 추가한 뒤에는
  `getComputedStyle()`로 실제 적용 여부를 확인하기 전까지 "규칙을 추가했으니 됐다"고
  믿지 말 것(스타일시트에 규칙이 존재하는 것과 그 규칙이 실제로 매칭되어 적용되는 것은
  별개다).**

## 8. 알려진 갭 — 이 문서가 못 채우는 부분

- 원본 소스(`spring-nexacro-N24-react`)와 `docs/migration/`의 코드 주석·README가
  `ADR-001`~`ADR-006`, `SEC.06`/`SEC.11` 같은 문서 번호를 인용하는데, 그 ADR/전략 문서
  자체는 이 저장소 어디에도 없다(검색 확인 완료). 아마 더 이전 단계의 상위 전략 저장소에
  있었을 것으로 추정되나, 지금은 참조가 깨진 상태다 — 화면 전환 작업 자체에는 지장이
  없지만, 나중에 "ADR-005가 정의한 형태"류 문구를 코드에 새로 추가할 때는 그 ADR이 실존
  하는지 먼저 확인할 것.
- Nexacro 런타임 콘솔에서 `nexacro.getApplication().langCode`를 직접 바꿔 언어를
  강제 전환해보려는 시도는 내부 상태 불일치로 에러가 났다(10200 검증 중 확인) — 언어별
  차이를 확인하려면 화면의 실제 언어 토글 버튼을 신뢰할 수 있는 방식으로 눌러야 하며,
  콘솔에서 상태를 직접 조작하는 건 피할 것.
